# Reflection

The hardest part of this project for me was understanding how to connect everything across multiple files. At the beginning, it was easy to write small pieces of code like a single function or class, but once the project grew, I had to think about how everything worked together. For example, I had to make sure that the Product, Vendor, and PurchaseOrder classes were all being used correctly in the inventory_manager file, and then make sure that the main menu could call those functions properly. Keeping track of where everything belonged and making sure I was importing the right things in each file was definitely the most challenging part.

The bug that took me the longest to solve was related to GitHub and pushing my project. I kept getting a 403 permission error, and it turned out that my computer was using the wrong GitHub account when trying to push the code. It was confusing because everything looked correct at first, but the credentials stored in my system were from a different account. I had to go into the Windows Credential Manager and remove the old GitHub login so I could sign in with the correct one. After that, the push finally worked. That was frustrating, but it taught me a lot about how Git and GitHub authentication actually work.

I organized my code by separating different responsibilities into different files. The models.py file contains all the classes like Product, Vendor, and PurchaseOrder. The inventory_manager.py file handles most of the logic, like adding, editing, searching, and sorting products, as well as creating and receiving purchase orders. The file_manager.py file is responsible for saving and loading data using JSON. The reports.py file contains all the report functions, like inventory reports and low stock reports. Finally, main.py controls the menu system and connects everything together. Breaking the project into multiple files made it easier to manage and understand.

My save and load system works by converting objects into dictionaries and then storing them in a JSON file. Each class has a to_dict method that turns the object into a format that can be saved. When loading the data, the program reads the JSON file and recreates the objects by passing the stored values back into the class constructors. This allows the program to keep data even after it is closed, which makes it more realistic and useful.

If I had another week to work on this project, I would improve the user interface and add more advanced features. Right now, everything runs in the console, which works but isn’t very user-friendly. I would also add more validation, like checking if a vendor exists before assigning it to a product, or making sure product IDs are always consistent. Another improvement would be adding better error messages and maybe even logging actions like when products are updated or purchase orders are received.

Overall, this project helped me understand how to build a larger program step by step. I learned how to organize code, work with multiple files, use classes effectively, and handle real-world concepts like inventory and ordering systems. I also gained experience with GitHub and version control, which was new to me. Even though there were some frustrating parts, I feel like I understand the project much better now and could explain how it works if asked.
